## Why

Story [P2.4](../../../docs/stories/p2-graphql-bff-expansion.md) continues the GraphQL migration after [P2.2](../../../docs/stories/p2-graphql-status-migration.md) (status on the server) and [P2.3](../../../docs/stories/p2-graphql-client-apollo.md) (Apollo on the dashboard). The operator console moves settings, mappings, activity, manual sync/reinit, integration nav, and dashboard setup detail (`dashboardSetup`, replacing `GET /api/ui/setup` on the web client) onto GraphQL and drops TanStack Query from `apps/web`. REST handlers stay registered until `P2.5` sunset. Moving those reads and writes to GraphQL aligns the console with one typed transport, enables deliberate cache policies and optimistic updates on CRUD-heavy screens, and avoids growing bespoke BFF REST endpoints per view.

The monorepo/React split in commit `2b71f909` also dropped material UX from the pre-split Handlebars pages: the Settings page lost its "Sync adapters" info card and per-field setting-key hints, the Mappings page lost inline row editing and the data-table layout (only `delete` survived), and the Activity page lost the per-run item tables, the triage action buttons (Review / Decline / Reconsider) with their optimistic UI, and the auto-poll-while-running behaviour. P2.4 already rewrites those three pages for Apollo and atomic layout, so restoring the missing UX in the same change is the cheapest path and scopes the atomic decomposition around the feature set those pages should carry going forward.

## What Changes

- **Server (Nest GraphQL, code-first):** New query fields — `effectiveSettings`, `mappings`, `activityFeed` (Relay-style cursor connection with `limit` / cursor args), `integrationNav` — plus mutations `updateSettings`, `upsertMapping`, `updateMapping`, `deleteMapping`, `triggerSync`, `reinitIntegration`, and triage mutations `acceptTriage`, `declineTriage`, `reconsiderTriage` bound to the same application logic behind the current `/api/triage/*` REST endpoints. Return shapes stay close to current REST payloads so the web client can swap transports with minimal shape churn.
- **Live updates (subscriptions, not polling):** Add an `activityEvents` subscription that delivers new/changed activity runs and items over the existing `graphql-ws` transport, bridged from the same domain signals as `statusUpdated` (sync start/end, per-item decision transitions, triage changes). The client does **not** poll — live refresh during a running sync cycle comes exclusively from subscriptions.
- **Web (`apps/web`):** Settings, Mappings, and Activity pages consume Apollo (`useQuery` / `useMutation` / `useSubscription`) instead of TanStack Query. Screen file layout follows the same **atomic-design** split as the dashboard: pages own data hooks and compose **templates** + **organisms**; no single monolithic page component for a migrated screen.
- **Restore pre-split UX dropped in commit `2b71f909`:**
  - **Settings:** bring back the "Sync adapters" info card (resolved source → destination labels, `VIBE_SQUIRE_SOURCE_TYPE` / `VIBE_SQUIRE_DESTINATION_TYPE` env-var references), per-field setting-key muted hints next to labels, and a save-success banner/toast indicating scheduler refresh when applicable. Server-side, `effectiveSettings` (or a sibling query) exposes field metadata (label, setting-key, env-var note, description) and the resolved adapter labels so the client does not hard-code them.
  - **Mappings:** restore the data-table layout with headers and inline **edit / save / cancel** per row (via a new `updateMapping` mutation). Surface VK repos fetch errors in a visible banner with the server message, and keep a reload/refetch control.
  - **Activity:** restore per-run expandable tables with item rows (PR #, title, author, repo, decision label, detail, kanban issue id) and the triage action buttons (Review / Decline / Reconsider) wired to the new triage mutations, with **optimistic responses** replacing the old `localStorage`-backed trick. Live updates come from `activityEvents` — no interval polling.
- **Apollo cache:** Explicit `typePolicies` / merge or read functions for list fields that mutations or subscriptions update (`mappings`, `activityFeed` connection, effective settings singleton) so list updates do not depend on manual `refetchQueries`. Optimistic responses for `updateSettings`, `upsertMapping`, `updateMapping`, `deleteMapping`, and the triage mutations.
- **Codegen:** Introduce `@graphql-codegen/cli` + `@graphql-codegen/client-preset` (or equivalent) and generated artifacts under `apps/web/src/__generated__/` once the operation surface exceeds trivial hand-typing — matching the P2.3 deferral note carried into P2.4.
- **Dependencies:** Remove `@tanstack/react-query` from `apps/web/package.json` when no remaining consumer needs it; if any screen stays on REST intentionally, document it in the PR and keep the dependency.
- **Tests:** Extend or add `apps/server/test/` integration coverage per new resolver and for the `activityEvents` subscription; existing REST integration tests remain passing.

## Capabilities

### New Capabilities

- `graphql-operator-bff`: GraphQL queries, mutations, and the `activityEvents` subscription backing the operator BFF domains (settings + field metadata, mappings CRUD with update, activity feed with cursor pagination and per-item triage, integration nav, sync control, triage accept/decline/reconsider). Resolvers remain thin adapters over existing application services (`settings`, `mappings`, activity API, triage, sync, `ui-nav`); Zod in `@vibe-squire/shared` stays validation source of truth with the same Zod↔GraphQL bridging pattern as P2.2.

### Modified Capabilities

- `web-graphql-client`: Beyond default `InMemoryCache()`, add named operations (or codegen-generated documents) for the new screens; custom cache field policies for mutated lists, settings, and the activity connection; optimistic mutation patterns (including triage); subscription-driven live refresh for activity (no client-side polling); optional removal of `QueryClientProvider` / TanStack Query when the last REST consumer in `apps/web` is gone; Vite proxy unchanged.
- `web-ui-atomic-design`: Atomic layout and import rules apply to **Settings**, **Mappings**, and **Activity** (and related sync/nav UI introduced in this migration), not only `DashboardPage` — pages stay the sole hook boundary; organisms receive data as props. Restored UX (Settings "Sync adapters" info card, Mappings editable table, Activity runs-with-items + triage actions) SHALL decompose into atoms/molecules/organisms rather than re-inlining into pages.

## Impact

- **Server:** `apps/server` GraphQL module and new resolver modules colocated with `settings`, `mappings`, `ui/activity-api`, `sync`, `ui-nav`, and triage (`sync/pr-triage.*`) services; `schema.graphql` / SDL grows; `activityEvents` subscription bridged from existing domain event signals following the `statusUpdated` pattern. Integration tests cover queries, mutations, and subscription.
- **Web:** `apps/web/src/pages/` thins for migrated routes; new or extended `ui/templates`, `ui/organisms`, `ui/molecules`, `ui/atoms` for restored UX (settings info card, editable mappings row molecule + mappings table organism, activity run/item organisms with triage-action molecule); `apps/web/src/graphql/operations/` (or codegen output) grows; `package.json` may drop TanStack Query. No client-side polling — all live updates ride `statusUpdated` and `activityEvents` subscriptions.
- **Shared:** Possible shared types or enum literals reused by new GraphQL types; no breaking change for external consumers (operator is co-deployed).
- **Not in scope:** REST/SSE sunset for status (`P2.5`); removal of REST controllers for these domains until explicitly scheduled; the legacy `localStorage` optimistic triage mechanism is replaced by Apollo optimistic responses and cleaned up.
