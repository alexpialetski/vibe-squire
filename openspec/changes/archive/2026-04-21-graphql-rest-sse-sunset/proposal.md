## Why

After P2.2–P2.4, the operator console (status, settings, mappings, activity, triage, dashboard setup, integration nav, manual sync) runs entirely on GraphQL. The duplicate REST/SSE surface — `GET /api/status`, `GET /api/status/stream`, `GET /api/ui/*`, the read-and-write side of mappings/settings, and the Vibe Kanban context endpoints consumed by `GithubPage` / `VibeKanbanPage` / VK repo loading in `MappingsPage` — exists only because the earlier stories chose not to migrate every consumer in one go. vibe-squire has exactly one user (the author), no external REST clients, and no backward-compatibility obligation. Leaving parallel transports in place forces every contributor (human or agent) to re-decide "which transport is authoritative?" every time they touch the server, keeps `@Sse`, `operator-bff.controller.ts`, `vibe-kanban-context.controller.ts`, and the DTO shims alive as dead weight, and blocks removal of `nestjs-zod` / `class-validator` / `class-transformer`. This change migrates the last three web REST holdouts to GraphQL, retires every REST/SSE surface that is not an explicit operator-tool keeper, and publishes a short transport policy so the decision stops re-litigating itself.

## What Changes

- **BREAKING**: Delete the SSE endpoint `GET /api/status/stream` and its `@Sse` handler. `StatusEventsService.updates()` feeds the GraphQL PubSub only.
- **BREAKING**: Delete every duplicate REST GET: `GET /api/status`, `GET /api/ui/setup`, `GET /api/ui/nav`, `GET /api/ui/settings-meta`, `GET /api/ui/github-fields`, `GET /api/settings`, `GET /api/mappings`, `GET /api/activity/runs`, and the entire `api/vibe-kanban/*` surface (`organizations`, `projects`, `repos`, `ui-state`).
- **BREAKING**: Delete REST mutation handlers that are GraphQL-superseded: `POST /api/mappings`, `PATCH /api/mappings/:id`, `DELETE /api/mappings/:id`, `PATCH /api/settings/core`, `PATCH /api/settings/source`, `PATCH /api/settings/destination`, and `POST /api/pr/{accept,decline,reconsider}`.
- **BREAKING**: Delete `apps/server/src/ui/operator-bff.controller.ts`, `apps/server/src/ui/activity-api.controller.ts`, `apps/server/src/vibe-kanban/vibe-kanban-context.controller.ts`, and the now-empty GET side of `apps/server/src/status/status.controller.ts`, `apps/server/src/settings/settings.controller.ts`, and `apps/server/src/mappings/mappings.controller.ts` (delete the files entirely if every handler is gone).
- **Kept as operator tools** (the only REST survivors): `POST /api/sync/run` and `POST /api/reinit`. Each is convenient from `curl` / `gh` / shell scripts for the single operator and has no GraphQL-subscription semantics worth porting.
- **Add new GraphQL operations** so the three web REST holdouts can move off `apiJson`:
  - `Query.githubFields: GithubFieldsPayload!` replaces `GET /api/ui/github-fields`.
  - `Query.vibeKanbanUiState: VibeKanbanUiState!` replaces `GET /api/vibe-kanban/ui-state`.
  - `Query.vibeKanbanOrganizations: [VibeKanbanOrganization!]!` replaces `GET /api/vibe-kanban/organizations`.
  - `Query.vibeKanbanProjects(organizationId: ID!): [VibeKanbanProject!]!` replaces `GET /api/vibe-kanban/projects?organization_id=…`.
  - `Query.vibeKanbanRepos: [VibeKanbanRepo!]!` replaces `GET /api/vibe-kanban/repos` (already partially covered; confirm).
  - `Mutation.updateSourceSettings(input: UpdateSourceSettingsInput!): EffectiveSettings!` replaces `PATCH /api/settings/source`.
  - `Mutation.updateDestinationSettings(input: UpdateDestinationSettingsInput!): EffectiveSettings!` replaces `PATCH /api/settings/destination`.
- **Migrate the web REST holdouts**: `GithubPage`, `VibeKanbanPage`, and VK repo loading in `MappingsPage` switch from `apiJson` to Apollo `useQuery` / `useMutation` against the operations above. After migration, `apps/web/src/api.ts` (the `apiJson` helper) is deleted.
- **Publish** `## Transport decision table` in `docs/ARCHITECTURE.md`. The status enum is reduced to `removed | kept (operator tool)` — there is no `kept (REST)` category because no REST endpoint survives for a non-operator reason.
- **Update** `AGENTS.md` and `.cursor/rules/architecture.mdc` to state the transport policy (GraphQL is authoritative; the only REST exceptions are the two operator-tool POSTs listed in the decision table) and link to the table as the source of truth.
- **Update** `README.md` API table: strip every removed row; leave `POST /api/sync/run` and `POST /api/reinit` (and add `POST /graphql` + `graphql-ws` subscriptions as first-class entries).
- **Delete** integration tests under `apps/server/test/` whose sole target is a removed endpoint; tests MUST NOT be left `.skip`ed or rewritten as no-ops. GraphQL integration specs already cover the retired behaviour; add tests for the new resolvers (GitHub fields, VK context, settings source/destination mutations).
- **Dependencies**: Remove `nestjs-zod`, `class-validator`, and `class-transformer` from `apps/server/package.json`. Delete the DTO files that became dead weight (`apps/server/src/mappings/dto/`, `apps/server/src/settings/dto/`, `apps/server/src/ui/dto/`, and any other DTO folder whose only consumers were removed handlers). Swagger decorators (`@ApiTags`, `@ApiOperation`, etc.) go along with the removed handlers; keep `@nestjs/swagger` only if the two surviving operator-tool endpoints still benefit from it — if not, drop it too.
- **Pre-flight gate**: Before deleting any endpoint, grep `apps/web/src` for `EventSource` and `apiJson` / `/api/`, grep `apps/server/test`, `README.md`, `docs/**`, and repo scripts, and confirm the only remaining callers of `apiJson` are the three pages being migrated in this same change. Every other hit aborts the commit until resolved.
- **Bundle-size check** after cleanup: confirm the dev-only Apollo Sandbox gating did not leak into prod, that removed controllers actually drop from the server bundle, and that `apps/web` bundle shrinks from dropping `apiJson` + unused Zod schemas.

## Capabilities

### New Capabilities

- `api-transport-policy`: Authoritative, documented rule that GraphQL (`/graphql` + `graphql-ws`) is the operator-console transport, `apiJson` is deleted, and the only REST survivors are the two operator-tool POSTs recorded in the `docs/ARCHITECTURE.md` decision table. Includes the decision-table format (status enum `removed | kept (operator tool)`), maintenance rules, and the `AGENTS.md` / `.cursor/rules/architecture.mdc` pointers.

### Modified Capabilities

- `graphql-status`: Replace the "REST and SSE status endpoints remain unchanged during the GraphQL migration" requirement with a requirement asserting that the REST `GET /api/status` controller and the SSE `@Sse('stream')` handler are deleted and that `StatusEventsService.updates()` feeds only the GraphQL PubSub.
- `graphql-operator-bff`: Reframe the parity scenarios that compare GraphQL responses to live REST endpoints (`effectiveSettings` vs `/api/ui/settings-meta`, `mappings` vs REST list, `activityFeed` vs REST activity, `integrationNav` vs REST aggregation, `dashboardSetup` vs `/api/ui/setup`) so they assert parity against the archived P2.4 contract and the backing store rather than a live REST endpoint. Add new requirements for `githubFields`, `vibeKanbanUiState`, `vibeKanbanOrganizations`, `vibeKanbanProjects`, `vibeKanbanRepos`, `updateSourceSettings`, and `updateDestinationSettings` — the operations that enable the web REST holdouts to move off `apiJson`.

## Impact

- **Server — deleted files (expected)**: `apps/server/src/ui/operator-bff.controller.ts`, `apps/server/src/ui/activity-api.controller.ts`, `apps/server/src/vibe-kanban/vibe-kanban-context.controller.ts`, every DTO under `apps/server/src/ui/dto/`, `apps/server/src/mappings/dto/`, `apps/server/src/settings/dto/` whose only consumer was a removed handler, and the related Zod parse-request wrappers. `status.controller.ts`, `settings.controller.ts`, `mappings.controller.ts`, and `pr-triage.controller.ts` are either deleted outright (if empty after handler removal) or slimmed to zero remaining handlers and then deleted.
- **Server — new resolvers**: GitHub fields query and Vibe Kanban context queries join the `GraphqlOperatorBffModule` (or an adjacent resolver module) as thin driving adapters over `SettingsService`, `VibeKanbanBoardService`, `SetupEvaluationService`, `UiNavService`. Source/destination settings mutations bind to `SettingsService.applyGroupPatch('source' | 'destination', input)`. Application services, ports, and Prisma schema are unchanged.
- **Server — kept REST**: `apps/server/src/sync/sync.controller.ts` (`POST /api/sync/run`) and `apps/server/src/reinit/reinit.controller.ts` (`POST /api/reinit`) stay. Both are single-handler controllers with no DTOs worth class-validating — they accept no body or a tiny Zod-parsed body.
- **Server — dependency removals**: `nestjs-zod`, `class-validator`, and `class-transformer` drop out of `apps/server/package.json`. `@nestjs/swagger` stays only if the two operator-tool POSTs keep Swagger docs; otherwise also removed.
- **Web — deleted**: `apps/web/src/api.ts` (`apiJson`) and any unused Zod response schemas it fed (e.g. references to `githubFieldsResponseSchema`, `vibeKanbanUiStateSchema` become unused once Apollo codegen types cover the same ground).
- **Web — rewritten**: `apps/web/src/pages/GithubPage.tsx`, `apps/web/src/pages/VibeKanbanPage.tsx`, `apps/web/src/pages/MappingsPage.tsx` (VK repo loading block only) switch to Apollo. Following the atomic-design rule the rewrite splits each page into presentational components under `apps/web/src/ui/` and keeps hooks at the page-level boundary (`apps/web/src/ui/README.md`).
- **Tests — server**: Delete `apps/server/test/*.integration-spec.ts` files whose only target is a removed endpoint. Add GraphQL integration specs for the new resolvers (GitHub fields disabled + enabled, VK context for orgs/projects/repos/ui-state, source/destination settings mutations hitting `SettingsService`).
- **Tests — web**: `GithubPage`, `VibeKanbanPage`, `MappingsPage` tests rewrite their fetch mocks as Apollo `MockedProvider` providers.
- **Docs**: `docs/ARCHITECTURE.md` gains the transport decision table; `AGENTS.md` and `.cursor/rules/architecture.mdc` reference it; `README.md` API table is pared to the two keepers plus the GraphQL entry.
- **No data migration**. Every behavioural change is at the HTTP boundary; no Prisma schema or SQLite migrations.
